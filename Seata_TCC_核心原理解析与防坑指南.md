# Seata TCC 模式核心原理解析与防坑实战指南

在分布式事务架构中，Seata 提供了多种模式。相比于基于数据库代理和 `undo_log` 的 AT 模式，TCC (Try-Confirm-Cancel) 模式是一种**完全依靠业务代码实现**的分布式事务模型。

本文档是对 TCC 核心概念、运行原理以及我们在开发过程中踩过的经典坑点的深度总结。

---

## 1. AT 模式 vs TCC 模式的本质区别

| 特性 | AT 模式 (Automated Transaction) | TCC 模式 (Try-Confirm-Cancel) |
| :--- | :--- | :--- |
| **接入成本** | **极低**。无代码侵入，靠一个 `@GlobalTransactional` 走天下。 | **较高**。必须手动编写 Try、Confirm、Cancel 三个阶段的核心逻辑。 |
| **底层原理** | **数据库层面**。代理拦截 SQL，自动生成 `undo_log` 反向补偿语句。 | **业务层面**。Seata 只负责编排流程，真正的补偿逻辑全靠你自己写的代码。 |
| **适用介质** | **极其局限**。仅支持具备本地 ACID 且支持 SQL 的关系型数据库（MySQL、Oracle）。 | **无所不能**。支持 Redis、MongoDB 等 NoSQL，也支持调用第三方 HTTP/RPC API接口甚至发邮件。 |
| **并发性能** | **较差**。存在分布式**全局行锁**，存在热点数据锁竞争瓶颈，极易超时卡死。 | **极高**。第一阶段 Try 立即释放本地数据库的行级锁，并发量有质的飞跃。 |

> **核心结论：**
> AT 模式是**无脑侵入数据库的自动化引擎**；TCC 是**抛弃自动化、纯手工接管的三段式火箭**。当并发要求极高（如电商扣库存）或涉及非数据库操作时，必须强上 TCC。

---

## 2. Try - Confirm - Cancel 到底在做些什么？

在 TCC 模式下，我们要实现三个方法。以“购买商品扣减库存”为例：

### 阶段一：Try (尝试 / 业务检查与资源预留)
* **核心目标**：检查库存够不够，如果够用，就把它“冻结”起来，防止别人抢走。
* **发生时间**：事务初期的正常单机 HTTP/RPC 调用期。
* **执行动作**：将真正的可用库存 (`residue`) 扣去相应数量，并塞进冻结库存 (`frozen`) 坑位里锁定。
* **数据库反应**：直接 Commit 提交了这条单机事务，**完全没有所谓的分布式死锁等待**，其他并发用户依然可以飞速操作剩下的可用 `residue`！

### 阶段二：Confirm (确认 / 真实业务提交)
* **发生时间**：所有参与者的 `Try` 均宣告成功后，Seata TC (事务协调器) 会统一号令全军执行 `Confirm`。
* **核心目标**：实打实地消耗掉当初预留的资源，不留痕迹。
* **执行动作**：由于在 Try 阶段已经扣掉过可用了，在这里只需要把账面上的那块冻结区块 (`frozen`) **抹掉清零**或者算入消耗总数 (`used`) 中即可。

### 阶段三：Cancel (取消 / 业务补偿与回滚)
* **发生时间**：只要军团里有任何一个微服务的 `Try` 报错失败了（例如下单失败、积分不足卡住了），Seata 会瞬间喊停，向所有刚刚 `Try` 成功的服务发送清退指令 `Cancel`。
* **核心目标**：把冻结的资源，原封不动地全部释放回去。
* **执行动作**：进入表里，将账户里的 `frozen` 扣除，并全额如数加还给可用库存 (`residue`) 里，实现物理还原。

## 3. TCC 绝不妥协的底线设计：幂等、防悬挂、空回滚

因为 TCC 需要借助网络发送 Confirm 和 Cancel 的重试指令，你的代码必须具备极强的防弹能力（防御性编程）：

### 为什么在 Confirm/Cancel 中绝不允许随意报错（也不能返回 False）？
如果不是数据库崩溃宕机这类硬件错误，而仅仅是因为业务层面逻辑的拦截（例如发现没收到过参数），**一定要放行并返回 `true`**。
如果你抛个异常，或者返回了 `false`，Seata TC 就会偏执地认为：“哦，网络抖动了被拒绝了，我必须一直重发这段指令直到它成功为止（**无限夺命连环重试**）！”

### 什么是空回滚？为什么遇到空参数必须 `return true`？
场景：在【订单】-【积分】-【库存】的分布式大事务中，积分 Try 成功，但是在调库的路上**网线断了**！
此时全局直接失败。Seata 开始发号施令，要求全员 Cancel 撤退。
此时你的库存微服务就会收到一张连 `productId` 参数都没有的 Cancel 空军令。
* **错误做法**：你以参数为空的理由抛了个错。结果就是 Seata 无限重试 Cancel 这个残次品。
* **正确做法（空回滚防御）**：发现核心参数缺失，说明老子连 `Try` 的皮毛都没见着，我怎么可能锁定过资源需要恢复？直接当着大家的面大喊一句：“我这边安全，无事发生”，然后 **`return true;`** 让 Seata 结案滚蛋。

### 幂等防御：执行过了就算成功
如果由于网络延迟，一个微服务连续接收到了两次同一个 ID 的 `Confirm` 指令，你要确保它只执行一次。
* **做法**：你的 SQL 需要带上前置判断条件，比如 `WHERE ... AND frozen >= #{count}`。如果执行完由于发现不够冻结金额而只更新了 0 行（说明之前已经被前置成功执行过清除了），这**不算失败**。这说明它早已成功！你应该直接 **`return true`**。

## 4. 经典的 Spring Boot & Seata 填坑实战纪实

### 惨烈报错："Cannot invoke "Object.toString()" because the return value of ... getActionContext is null"
* **案发现场**：控制台中疯狂刷屏，每秒打印一次 Confirm 阶段参数取到了 null，无限次疯狂重试导致微服务完全瘫痪死锁。
* **凶手**：Spring Boot 的 CGLIB 代理机制。
* **案发原理**：
  在早期的代码中，我们将 Seata 专用的拦截注解 `@BusinessActionContextParameter` 全都挂载了 Java 的 `interface` 上（如 `StorageTccService`）。
  但在运行期中，Spring AOP 生成的代理类 (`StorageTccServiceImpl$$SpringCGLIB...`) 优先拦截的却是其实现类。由于实现类的形参上当时忘记打上此注解，Seata 的切面没能抓取到 `productId` 等实参，使得 Try 产生了一个“干瘪、残疾”的历史僵尸记录，并长久地被塞入了 SeataTC 的硬盘里面。
* **修复方法**：
  1. 保证 `StorageTccServiceImpl` (实现类) 的 `tryDecrease` 对应参数上也要标注 `@BusinessActionContextParameter`。（切记一定要引入包 `import` 而不是直接写臭长丑陋的全限定类名）
  2. 为了终止早前已经记录到 TC 的那批怎么重试都会空指的无限死循环残渣请求，我们直接在 `Confirm` 中强制加入硬拦截：“如果发现 `null` 这种残次品，不报错，而是假装成功直接 `return true`，好糊弄 Seata TC 删掉那条残次品！”。在这操作之后，控制台顺利安静了下来。
